# 小游戏设计能力规则

本项目的生成器必须先选择游戏类型，再把主题素材写入受约束的 `game_spec.json`。不要让模型或 fallback generator 直接生成任意 JavaScript。

## 类型选择

- `choice_adventure`：分支选择、剧情、结局、冒险关键词优先。
- `quiz`：问答、测验、答题、知识挑战关键词优先，至少 5 道题。
- `clicker`：点击、收集、分数、连击关键词优先，必须有移动或随机目标、奖励和干扰物。
- `memory`：记忆、翻牌、配对关键词优先，必须有多组可配对卡牌和步数限制。
- `dodge`：躲避、障碍、生存、飞船关键词优先，必须有开始按钮、倒计时和逐步增加的障碍。
- `escape_room`：密室、逃脱、谜题、找线索关键词优先，必须有线索收集、答案输入和失败提示。

## 标题和简介

- 标题不能直接照搬用户 prompt。
- 同一主题下不同玩法必须有不同标题后缀和简介，例如“问答挑战”“点击收集”“躲避生存”。
- 简介必须描述玩家行为，不只是复述“主角将在几个场景完成目标”。

## 标签

标签来自固定 taxonomy，并控制在 3 到 6 个主要标签展示：

- 类型：互动剧情、分支选择、问答、解谜、点击、收集、记忆、配对、动作、躲避、密室逃脱、冒险、生存、轻度动作。
- 题材：魔法、森林、奇幻、赛博、城市、校园、考试、太空、飞船、科幻、海盗、宝藏、童话、悬疑。
- 风格：可爱、黑暗、治愈、搞笑、像素、未来感。
- 难度：简单、中等、困难。
- 时长：1 分钟、3 分钟、5 分钟。

非互动剧情类型不要自动打上“互动剧情”。例如点击收集应使用“点击、收集、轻度动作”，躲避游戏应使用“动作、躲避、生存”。

## 可玩性底线

- 每类游戏都必须有明确胜负条件、反馈文案和重新开始能力。
- Dodge 不能进入即失败，必须由玩家点击开始后才运行。
- Clicker 不能只有一个大按钮，必须出现随机目标或移动目标。
- Quiz 不能少于 5 题，且每题有解释。
- Play 页必须保持 sandbox iframe 加载远端 manifest/entry，不能改为硬编码本地组件。
